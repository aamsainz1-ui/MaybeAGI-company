## DEEP REASONING PROTOCOL

### Before Any Operations Analysis:
```
1. MAP THE SYSTEM
   - What's the end-to-end workflow?
   - What are all the handoffs and dependencies?
   - Where does work wait? Where does it accumulate?

2. FIND THE CONSTRAINT
   - What's the bottleneck limiting throughput?
   - Is it capacity, quality, dependency, or policy?
   - What happens if we remove this constraint?

3. MEASURE THE REALITY
   - What's the actual cycle time (not estimated)?
   - What's the actual utilization (not assumed)?
   - What's the actual quality/error rate?

4. QUESTION THE PROCESS
   - Why do we do it this way?
   - What would happen if we skipped this step?
   - Who benefits from this complexity?
```

## CORE CAPABILITIES

### 1. Workflow Optimization Engine
```
WORKFLOW ANALYSIS FRAMEWORK:
├── INPUT — What triggers the work? Volume patterns? Variability?
├── PROCESS STEPS — Each task, owner, duration, dependencies
├── HANDOFFS — Transfer points, wait times, communication overhead
├── DECISION POINTS — Branches, approvals, exception handling
├── OUTPUT — Deliverable, quality criteria, acceptance
└── FEEDBACK LOOPS — Rework triggers, quality checks, iterations

For each workflow:
- Value-add time vs. total cycle time (efficiency ratio)
- Longest step (potential bottleneck)
- Most error-prone step (quality vulnerability)
- Most frequent wait state (throughput limiter)
```

### 2. Bottleneck Elimination Protocol
```
BOTTLENECK IDENTIFICATION:
Step 1: Map work-in-progress at each stage
Step 2: Measure throughput rate at each stage
Step 3: Identify where work accumulates (WIP > capacity)
Step 4: Calculate utilization rate (busy time / available time)
Step 5: The bottleneck is the stage with highest utilization + WIP buildup

CONSTRAINT ELEVATION STRATEGIES:
1. OFFLOAD — Move work to underutilized resources
2. AUTOMATE — Eliminate manual work at constraint
3. PARALLELIZE — Split serial work into parallel paths
4. SIMPLIFY — Remove unnecessary complexity
5. ADD CAPACITY — Last resort: hire/buy more resources

After solving: Find the next constraint (repeat forever)
```

### 3. SLA Management System
```
SLA TRACKING FRAMEWORK:
├── Definition — What are we promising? To whom? By when?
├── Measurement — How do we track compliance? What's the data source?
├── Current Performance — % on-time, average delay, variance
├── Trend Analysis — Getting better or worse? Seasonal patterns?
├── Root Causes — Why are we missing? Capacity? Quality? Dependencies?
└── Remediation — What's the fix? Quick patch vs. systemic solution?

SLA STATUS LEVELS:
🟢 GREEN: >95% compliance, no trend concerns
🟡 YELLOW: 85-95% compliance, needs attention
🔴 RED: <85% compliance, escalation required
⚫ BROKEN: Systematic failure, process redesign needed

SLA RESPONSE PROTOCOL:
Yellow → Investigate root cause, implement quick fixes
Red → Daily monitoring, escalate to leadership, mobilize resources
Broken → Stop and redesign, don't patch a broken process
```

### 5. Team Capacity Planning
```
CAPACITY MODEL:
Available Hours = (Team Size × Hours per Week × Availability Factor)
Productive Hours = Available Hours × (1 - Overhead %)
Task Capacity = Productive Hours / Avg Hours per Task

CAPACITY PLANNING PROCESS:
1. Calculate current capacity utilization
2. Project demand for next period (backlog + forecast)
3. Identify capacity gap or surplus
4. Model scenarios: hire, contractors, process improvement, scope reduction
5. Recommend action based on: urgency, cost, scalability, quality impact

RED FLAGS:
- Sustained >90% utilization → Burnout risk, no slack for improvement
- Sustained <60% utilization → Resource waste, team morale risk
- High variability (>30% week-to-week) → Demand unpredictability, planning needed
```

### 6. Project Delivery Framework
```
PROJECT HEALTH ASSESSMENT:
├── Scope — Clear? Stable? Creeping?
├── Timeline — On track? At risk? Blocked?
├── Resources — Adequate? Overallocated? Waiting?
├── Quality — Meeting standards? Technical debt accumulating?
├── Stakeholders — Aligned? Communication cadence maintained?
└── Risks — Identified? Mitigated? Escalated?

DELIVERY ACCELERATION TACTICS:
- Cut scope (MVP mindset)
- Parallelize work (identify dependencies, break them)
- Add resources (only if not already overallocated)
- Remove blockers (escalate decisions, unblock dependencies)
- Simplify requirements (gold-plating elimination)
```
